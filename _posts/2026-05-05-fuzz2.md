# Fuzzing101 실습 분석: Exercise 1 & 2 & 4

Fuzzing101 실습 중 **Exercise 1 (Xpdf)**과 **Exercise 2(libexif), **Exercise 4 (LibTIFF)**를 통해 서로 다른 유형의 취약점을 분석하고 탐지하는 과정을 정리합니다.

## 1. Exercise 1: Xpdf (CVE-2019-13288)

① **개인별 환경 구축**
- 컴파일러: afl-clang-fast를 사용하여 소스 코드 레벨의 Instrumentation을 적용했습니다.
- 도구: AFL++ v4.0 이상 환경에서 pdftotext 바이너리를 빌드했습니다.

② **분석 과정**
1. **대상 선정**: PDF 파일을 텍스트로 변환하는 `pdftotext` 유틸리티. & PDF 문서에서 텍스트를 추출하는 Xpdf 4.01.01.
2. **Instrumentation**: `afl-clang-fast` 컴파일러를 사용하여 코드 실행 경로를 추적할 수 있도록 바이너리를 빌드했습니다.
3. **시드(Seed) 설정**: 퍼징의 시작점이 될 유효한 샘플 PDF 파일들을 준비했습니다.
   
특징 : 복잡한 트리구조를 가진 PDF 오브젝트를 파싱할때 재귀함수를 빈번하게 호출하는 구조라는 점을 확인했습니다.

③ **Fuzzing 전략 수립 및 실행**
- 전략: 유효한 샘플 PDF 파일들을 시드(Seed)로 설정하고, 변이 기반 퍼징을 수행했습니다.
- 명령어: afl-fuzz -i in -o out -- ./pdftotext @@

  멈춤 현상을 확인했습니다.
* **취약점 원인**: 분석 결과, 특정 PDF 구조가 함수를 계속해서 호출하게 만드는 **무한 재귀(Infinite Recursion)**를 유발했습니다.

## 2. Exercise 2: LibxiF (CVE-2009-3895 & CVE-2012-2836)

### 2.1 ASan 기반 정밀 탐지

① **개인별 환경 구축**
- 탐지 도구: 단순 크래시로 나타나지 않고 무한히 자원을 소모하는 'Hang' 현상을 포착하기 위해 AFL의 타임아웃 감시 기능을 활용했습니다.
- 도입 배경: 서비스 거부(DoS) 공격 중 일부는 프로그램을 즉시 종료시키지 않고, 무한 루프를 유발하여 시스템 가용성을 저해합니다. 이를 위해 AFL Hang 탐지를 통해 비정상적으로 실행 시간이 긴 케이스를 추출했습니다.
- 설정: 표준 AFL 퍼징 환경에서 타겟 바이너리 tiff2pdf를 빌드하여 논리적 루프 발생 여부를 감시하도록 구성했습니다.

② **타겟 분석 및 사전 리서치**
- 대상: TIFF 이미지를 PDF로 변환하는 tiff2pdf (libtiff 4.0.2).
- 특징: TIFF 파일은 이미지 페이지 정보를 IFD(Image File Directory)라는 체인 구조로 관리합니다. 각 IFD는 다음 IFD의 위치를 가리키는 'Next IFD Offset' 포인터를 가지며, 라이브러리는 이를 따라 페이지를 탐색합니다.
- 취약점 가설: 데이터 구조의 연결성(Chaining)을 검증 없이 신뢰할 경우, 포인터가 이전 지점을 다시 가리키는 순환 참조(Circular Reference)가 발생하여 무한 루프에 빠질 가능성이 높다고 판단했습니다.

③ **Fuzzing 전략 수립 및 실행**
- 전략: 비정상적인 입력값에 의한 프로그램 정지(Hang) 상태를 집중적으로 수집하는 전략을 채택했습니다.
- 명령어: afl-fuzz -i in -o out -- ./tiff2pdf -o /dev/null @@

④ **Crash 분석 및 취약점 원인 파악**

- 현상: 특정 입력 파일(id:000000) 처리 시 프로그램이 리턴되지 않고 무한히 실행되며 CPU 점유율 100%를 유지하는 현상이 탐지되었습니다.
  
- 결과 (CVE-2012-3401): TIFF 파일 내부의 IFD 엔트리가 끝(NULL)을 가리키지 않고 다시 자기 자신이나 상위 엔트리를 가리키도록 조작되어       Infinite Loop가 발생합니다.

- 상세 분석: * GDB 분석 결과: tif_dir.c의 TIFFSetDirectory() 함수에서 인자값 dirn(디렉토리 번호)이 1 → 2 → 3 ... 순으로 중단 없이 무한   히 증가하며 호출되는 것을 확인했습니다.
 
- 코드 레벨 원인: tiff2pdf는 페이지를 넘길 때마다 nextdir 오프셋을 읽어 이동하는데, 조작된 파일은 이 오프셋이 물리적으로 이전 위치를 가     리키게 하여 논리적 끝을 찾지 못하게 만듭니다.

근본 원인: 라이브러리가 다음 오프셋으로 이동할 때 "이미 방문한 오프셋인가?"를 확인하는 검증 로직이 부재하여, 논리적으로 끝이 없는 페이지 체인을 무한히 탐색하는 설계적 결함이 존재함을 특정했습니다.

## 3. Exercise 4: LibTIFF (CVE-2016-9243)

① **개인별 환경 구축**

- 대상 라이브러리: tiff-4.0.4
- OS: Ubuntu (WSL2)
- Compiler: afl-clang-fast (AFL++)
- ASan 설정: export AFL_USE_ASAN=1 옵션을 적용하여 빌드.

단순 크래시뿐만 아니라 힙(Heap) 영역의 메모리 오염(Overlap, Overflow 등)을 정밀하게 추적하도록 구성했습니다.
  
② **분석 과정 및 사전 리서치**

1. **대상 선정**: TIFF 이미지를 PDF로 변환하는 `tiff2pdf` 도구.
2. **ASan 적용**: 단순한 크래시뿐만 아니라 보이지 않는 메모리 오류를 잡아내기 위해 **AddressSanitizer(ASan)**를 활성화하여 컴파일했습니다. - `export AFL_USE_ASAN=1` 설정 후 빌드 진행.
3. **탐지 메커니즘**: 메모리 접근을 실시간으로 감시하여, 할당된 범위 밖을 건드리는 순간 즉시 탐지하도록 설정했습니다.
4. **코드 분석**: 이미지 데이터 복사 및 포맷 변환 과정에서 대량의 메모리 할당과 memcpy 연산이 빈번하게 발생하여 메모리 관련 취약점 발생 가능성이 높음을 확인했습니다.

③ **Fuzzing 전략 수립 및 실행**

- 전략: 유효한 TIFF 샘플을 입력값(In-house Seed)으로 사용하고, 반복 변이를 통해 힙 메모리 할당 로직을 집중적으로 퍼징했습니다.

- 명령어: afl-fuzz -i in -o out -m none -- ./tiff2pdf @@

### 버그 발견 및 결과 분석
* **탐지된 취약점**: memcpy-param-overlap
* **탐지 방법**: 일반 퍼징으로는 발견하기 힘든 미세한 메모리 오염을 ASan 로그를 통해 포착했습니다.
* **취약점 원인**: tiff2pdf.c의 3563번 라인 부근에서 복사 소스(Source)와 목적지(Destination) 메모리 영역이 겹치는 현상이 발생했습니다.
* **상세 분석**: _TIFFmemcpy 함수를 사용했으나, 복사하려는 두 영역이 중첩될 경우 memcpy는 정의되지 않은 동작을 유발합니다. 이는 메모리 오염 및 예기치 못한 프로그램 종료의 원인이 됩니다.
* 
⑤ 패치 및 검증 (Patch & Verification)
분석된 취약점을 해결하기 위해 소스 코드를 수정하고 패치 여부를 검증했습니다.

1. 소스 코드 수정 (Patch)
중첩된 메모리 영역에서도 안전하게 동작하는 memmove 함수로 교체하였습니다.

`$ diff -u tiff2pdf.c.bak tiff2pdf.c`

<img src="assets/img/fuz4add1.png" alt="퍼즈 실습환경" width="500">

2. 재빌드 및 최종 검증
수정된 소스를 바탕으로 make clean && make를 실행하여 바이너리를 다시 생성한 후, 동일한 크래시 파일을 입력값으로 주어 결과를 확인했습니다.
`$ /home/rudql/fuzzing_tiff/tiff-4.0.4/tools/tiff2pdf -o /dev/null id:000000*`

<img src="assets/img/fuz4add3.png" alt="퍼즈 실습환경" width="500">

[최종 결과]
패치 후 실행 결과, 이전의 AddressSanitizer: memcpy-param-overlap 에러가 더 이상 발생하지 않음을 확인했습니다. 비정상적인 데이터에 대해 프로그램이 크래시 없이 안전하게 에러 처리를 수행하며 종료되는 것을 확인하여 패치 성공을 증명했습니다.

이번 실습을 통해 AFL++와 ASan을 연동하여 눈에 보이지 않는 메모리 중첩 및 오염 취약점을 찾는 과정을 학습했습니다. 특히 발견된 취약점을 분석하고 memmove를 이용해 직접 패치 및 재검증을 수행함으로써, 보안 취약점의 발견부터 조치까지의 전체 사이클을 경험할 수 있었습니다.
