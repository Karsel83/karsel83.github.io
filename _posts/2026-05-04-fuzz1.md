## 1. 퍼징(Fuzzing)이란?
퍼징은 소프트웨어의 보안 약점을 찾기 위해 **변조된 데이터(Fuzz)**를 대상 프로그램에 무작위로 입력하여, 프로그램이 예상치 못한 방식으로 작동하거나 **비정상적으로 종료(Crash)**되는 지점을 찾아내는 자동화된 보안 테스트 기법입니다. 수동으로 찾기 어려운 메모리 오버플로우나 논리적 에러를 대량의 데이터를 통해 전수 조사하듯 탐색하는 것이 특징입니다.

---

## 2. 퍼징의 종류 (내부 정보 활용도에 따른 분류)

퍼저는 분석 대상의 내부 구조를 얼마나 파악하고 있느냐에 따라 세 가지로 나뉩니다.

### 1) Black-box Fuzzing (블랙박스 퍼징)
* **특징**: 프로그램의 내부 소스 코드나 구조를 전혀 모르는 상태에서 입력을 생성합니다.
* **장점**: 설정이 간단하며 프로그램 종류에 상관없이 적용 가능합니다.
* **단점**: 코드의 깊은 부분(Deep Logic)까지 도달하기 어렵고 효율성이 낮습니다.

### 2) White-box Fuzzing (화이트박스 퍼징)
* **특징**: 소스 코드를 완전히 분석하여 모든 실행 가능한 경로를 탐색합니다.
* **장점**: 취약점이 존재할 만한 복잡한 경로를 정밀하게 추적합니다.
* **단점**: 분석 오버헤드가 매우 커서 탐색 속도가 매우 느립니다.

### 3) Grey-box Fuzzing (그레이박스 퍼징)
* **특징**: 프로그램 내부 구조를 일부 파악하며, 컴파일 시 '계측(Instrumentation)' 코드를 삽입해 실행 경로에 대한 피드백을 받습니다.
* **장점**: 블랙박스보다 효율적이고 화이트박스보다 빠릅니다.
* **사례**: 현재 가장 널리 사용되는 **AFL++**가 이 방식에 해당합니다.

---

## 3. 주요 퍼저(Fuzzer) 비교 분석

현재 보안 연구 및 산업 현장에서 주로 사용되는 퍼저들의 특징과 차이점입니다.

| 구분 | AFL++ | Jackalope | LibFuzzer |
| :--- | :--- | :--- | :--- |
| **핵심 개념** | Grey-box / Coverage-guided | Black-box & Grey-box 지원 | In-process 퍼징 |
| **주요 특징** | 다양한 변이 알고리즘 탑재 및 강력한 커뮤니티 지원. | Windows/macOS 등 클로즈드 소스 환경 분석에 최적화. | 특정 함수를 직접 호출하여 분석하므로 실행 속도가 압도적임. |
| **운영체제** | 주로 Linux (WSL 포함) | Windows, macOS, Linux | 플랫폼 무관 (라이브러리 단위) |
| **차이점** | **범용성**: 일반적인 바이너리 취약점 탐색에 강점. | **유연성**: 소스 코드가 없는 상용 소프트웨어 분석에 유리. | **정밀도**: 특정 API나 라이브러리 내부를 정밀 분석할 때 유리. |

## 4. 퍼징의 핵심 메커니즘: 시드(Seed)와 변이
퍼징은 준비된 입력값(Seed)을 변형하여 프로그램의 예외 상황을 유도합니다. 시드를 변형하는 방식은 크게 두 가지로 나뉩니다.
* **Random**: 입력 데이터를 무작위로 변경합니다.
* **Confusion**: 데이터의 구조나 의미를 혼란스럽게 만드는 정교한 변이를 가합니다.

변이된 입력값(Mutation)을 타겟 프로그램에 주입한 후, 출력(Output) 결과나 시스템의 반응을 확인하여 취약점을 식별합니다.

---

## 5. Grey-box 퍼징과 커버리지(Coverage)
AFL++와 같은 현대적인 퍼저는 단순히 데이터를 던지는 것에 그치지 않고 **코드 커버리지**를 측정합니다.
* **원리**: 입력을 넣었을 때 프로그램 내부의 다음 상태(State)나 코드 경로를 얼마나 새롭게 찾아냈는지 기록합니다.
* **AFL의 특징**: 타겟 프로그램의 소스 코드를 컴파일할 때 계측 코드를 삽입하여, 실행 시마다 커버리지를 피드백 받아 더 효율적인 변이를 생성합니다.

---

## 6. 타겟별 퍼징 전략: 네트워크와 문서
분석 대상의 특성에 따라 사용하는 도구와 전략이 달라집니다.
* **문서 파일(PDF, HWP)**: 주로 AFL++를 사용하며, 복잡한 파일 형식을 파싱하는 로직을 집중적으로 분석합니다.
* **네트워크**: **부퍼즈(BooFuzz)**와 같은 도구가 주로 사용됩니다.
    * **헤더 고정**: 카메라 분석 등 네트워크 프로토콜을 퍼징할 때, 변하지 않는 헤더 부분은 고정하고 데이터 필드만 변이시켜 효율을 높입니다.

---

## 7. 하네스 코드(Harness Code)의 중요성
퍼징에서 가장 중요한 작업 중 하나는 **하네스 코드** 작성입니다.
* **역할**: 타겟 프로그램의 핵심 동작을 퍼저가 호출하기 쉬운 형태로 구현한 일종의 "시험대"입니다.
* **예시**: PDF 분석 기능을 퍼징한다면, 전체 프로그램을 다 돌리는 대신 PDF 파싱 엔진만 별도로 떼어내어 동작하게끔 하네스를 설계해야 퍼징 속도와 효율이 극대화됩니다.

---

## 8. 최신 퍼징 기술 트렌드

### 하이브리드 퍼징 (Hybrid Fuzzing)
복잡한 조건문(예: 특정 문자열 비교)은 일반적인 퍼징으로 뚫기 어렵습니다. 이를 해결하기 위해 **심볼릭 실행(Symbolic Execution)**을 결합합니다.
* **Z3 Solver**: 복잡한 조건을 수식(방정식)으로 세워 빠르게 해를 구합니다.
* **운용 방식**: AFL++로 퍼징을 수행하다가 커버리지가 더 이상 늘어나지 않으면, 심볼릭 실행기를 투입해 막힌 조건문을 풀고 다시 퍼징을 이어갑니다.

### ChatAFL 및 LLM 결합
최근에는 **LLM(Large Language Model)**을 퍼징에 결합하려는 시도가 활발합니다.
* **코드 자동화**: 복잡하고 번거로운 하네스 코드 작성을 LLM이 대신 수행하거나, 프로그램의 의미를 분석하여 더 효과적인 시드를 생성하도록 돕습니다.

## 10. 실습한 결과

본 실습은 오픈 소스 이미지 처리 라이브러리인 **LibTIFF(v4.0.4)**의 `tiff2pdf` 도구를 대상으로 퍼징(Fuzzing)을 수행하여 잠재적인 메모리 취약점을 발견하고, 이를 재현 및 분석하는 것을 목적으로 한다.

# 타겟 정보
*   **Target:** `tiff2pdf` (LibTIFF 4.0.4)
*   **Vulnerability:** CVE-2016-9532 (Heap-based buffer overflow / memcpy-param-overlap)

# 빌드 환경
AFL++ 인스트루먼테이션과 AddressSanitizer(ASan)를 활성화하여 메모리 오류를 즉각 탐지할 수 있도록 빌드하였다.

# ASan 활성화 빌드 설정

```bash
./configure --enable-shared=no CC=afl-clang-fast CXX=afl-clang-fast++

export AFL_USE_ASAN=1

make -j$(nproc)

# 퍼징 단계별 전략

* Initial Seed: 유효한 소형 TIFF 샘플 파일을 입력으로 사용.

* Mutation: AFL++의 havoc, splice 알고리즘을 활용한 입력값 변조.

* Monitoring: 약 10시간 동안 실시간 대시보드를 통해 실행 속도(exec speed) 및 고유 경로(new edges) 탐색 상태를 확인.

여기서 메모리 오류를 탐지할 수 있었는데, memcpy 함수 호출 시 소스와 목적지 메모리 영역이 겹치는 현상이 발생함. 이는 프로그램이 이미지 데이터를 처리하는 과정에서 버퍼의 크기나 위치를 잘못 계산하고 있음을 의미합니다.

본 실습을 통해 LibTIFF 4.0.4 버전의 tiff2pdf에서 발생하는 **CVE-2016-9532 취약점을 성공적으로 재현**하였다.

* 패치 적용: 상위 버전에서는 memcpy 대신 메모리 중첩을 허용하는 memmove를 사용하거나, 버퍼 경계 검사를 강화하는 패치가 적용되어 있다.

* 퍼징의 유용성: 수동 분석으로는 찾기 힘든 예외 케이스를 자동화된 퍼징 도구(AFL++)를 통해 단시간 내에 다수 확보할 수 있음을 확인하였다.
