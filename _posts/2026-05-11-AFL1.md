## 0. 목표
이 보고서의 목표는 AFL, 즉** American Fuzzy Lop**의 핵심 동작 원리를 이해하는 것이다.  
특히 다음 네 가지를 중심으로 분석한다.

1. Instrumentation
2. Coverage
3. Fork Server
4. Mutation

주의할 점은 이 보고서가 AFL++가 아니라 **google/AFL 원본 구현**을 기준으로 한다는 점이다.

AFL은 단순히 랜덤 데이터를 계속 던지는 퍼저가 아니다. 
AFL README는 AFL을 “instrumentation-guided genetic algorithm”을 사용하는 brute-force fuzzer라고 설명한다.  
즉, AFL은 입력을 변형하면서 프로그램을 실행하고, 그 입력이 새로운 실행 경로를 만들었는지 관찰한 뒤, 의미 있는 입력만 다시 queue에 저장한다.

AFL의 전체 구조는 다음 네 가지로 나눠서 볼 수 있다.
```
1. Instrumentation
   → 타겟 프로그램 안에 coverage 기록 코드를 삽입한다.

2. Coverage
   → 실행 중 기록된 bitmap을 보고 새로운 경로인지 판단한다.

3. Fork Server
   → 매번 execve()를 반복하지 않고 fork()로 빠르게 실행한다.

4. Mutation
   → 기존 seed를 변형해 새로운 경로를 만들 입력을 찾는다.
```

AFL의 기본 흐름은 다음과 같다.
```
초기 seed 입력 준비
        ↓
afl-fuzz가 seed를 선택
        ↓
입력을 변형 mutation
        ↓
target program 실행
        ↓
instrumentation 코드가 coverage bitmap 기록
        ↓
afl-fuzz가 bitmap 확인
        ↓
새로운 coverage가 있으면 queue에 저장
        ↓
반복
```

## 1.Instrumentation — AFL은 프로그램을 어떻게 관찰하는가?
AFL에서 instrumentation은 **타겟 프로그램의 실행 흐름을 관찰하기 위해 컴파일 과정에서 추적 코드를 삽입하는 것**이다.

일반적인 프로그램은 실행 중 어떤 branch를 지났는지 외부에서 알 수 없다. AFL은 이 문제를 해결하기 위해  
` afl-gcc` , `afl-clang` 같은 wrapper를 제공한다.  
source code가 있는 경우 기존 `gcc`나 `clang` 대신 이 wrapper로 컴파일하면 instrumentation이 포함된 binary가 만들어진다.
AFL README에서도 source가 있는 경우 `afl-gcc` 나 `afl-clang` 을 drop-in replacement처럼 사용한다고 설명한다.  

google/AFL 원본에서 중요한 파일은 다음과 같다.
```
afl-gcc.c  → gcc/clang wrapper
afl-as.c   → assembly를 읽고 instrumentation 삽입
afl-as.h   → 삽입되는 trampoline, forkserver, logging payload 정의
config.h   → MAP_SIZE, FORKSRV_FD, 환경변수 등 설정
```

AFL의 instrumentation 흐름은 내가 분석했을 때에는 다음과 같다.
```
사용자:
    ./afl-gcc target.c -o target

내부:
    afl-gcc
        ↓
    gcc 실행 옵션 수정
        ↓
    gcc가 assembly 생성
        ↓
    afl-as가 assembly를 가로채서 수정
        ↓
    branch 주변에 trampoline 삽입
        ↓
    실제 assembler가 object file 생성
        ↓
    instrumented binary 완성
```

afl-gcc.c는 실제 컴파일러가 아니라 wrapper이다. 내부적으로 `-B` 옵션을 추가해 gcc가 기본 assembler 대신 AFL의 assembler wrapper를 먼저 사용하게 만든다.
clang 모드에서는 integrated assembler를 비활성화하기 위해 `-no-integrated-as`도 추가한다.

실제 instrumentation 삽입은 `afl-as.c`의 `add_instrumentation()`에서 수행된다. 이 함수는 assembly를 한 줄씩 읽으면서 .text section, branch label, conditional jump 등을 찾아 trampoline을 삽입한다. `afl-as.c` 주석에는 AFL이 잡으려는 예시로` main:`, `.L0:`, `.LBB0_0:`, `jnz foo` 같은 지점이 나온다.

여기서 trampoline은 타겟 프로그램 중간에 삽입되는 작은 assembly 코드 조각이다. 이 코드는 현재 위치 ID를 저장한 뒤 `__afl_maybe_log`를 호출한다.

개념적으로는 다음과 같다.
```
cur_location = <COMPILE_TIME_RANDOM>;
shared_mem[cur_location ^ prev_location]++;
prev_location = cur_location >> 1;
```
AFL technical details 문서에서도 instrumentation이 branch edge coverage와 coarse hit count를 기록하며, 삽입된 코드가 위 pseudocode와 본질적으로 같다고 설명한다.

여기서 핵심은` cur_location ^ prev_location`이다. AFL은 단순히 “어느 basic block에 도달했는가”가 아니라,  
**이전 위치에서 현재 위치로 이동한 edge를 기록**한다.

## 2. Coverage — AFL은 새로운 경로를 어떻게 판단하는가?

Instrumentation이 정보를 기록하는 장치라면, coverage는 그 정보를 해석하는 기준이다.

AFL은 실행 결과를 64KB shared memory bitmap에 기록한다.  
`config.h`에는 MAP_SIZE_POW2가 16으로 정의되어 있고, `MAP_SIZE`는` 1 << MAP_SIZE_POW2`로 계산된다. 따라서 기본 bitmap 크기는 `2^16 = 65536 bytes`, 즉 64KB이다.

이 bitmap은 `afl-fuzz`와 target program이 공유한다. AFL은 shared memory ID를 환경변수 `__AFL_SHM_ID`로 target에 전달한다. 이 값도 `config.h`에 정의되어 있다.

실행 중 구조는 다음과 같다.

```
afl-fuzz
  └─ shared memory 생성
        ↓
target program 실행
        ↓
instrumentation 코드가 bitmap 갱신
        ↓
target 종료
        ↓
afl-fuzz가 bitmap 분석

```
AFL technical details 문서에서는 `shared_mem[]`이 instrumented binary에 전달되는 64KB SHM 영역이며, output map의 각 byte는 `(branch_src, branch_dst)` edge tuple의 실행 흔적을 나타낸다.

AFL은 bitmap을 보고 다음을 판단한다.

1. 완전히 새로운 edge가 나왔는가?
2. 이미 본 edge지만 hit count가 달라졌는가?
3. 아무 변화도 없는가?

이 판단에 중요한 함수가 afl-fuzz.c의 `has_new_bits()`이다. 주석에 따르면 이 함수는 현재 실행 경로가 새로운 정보를 가져왔는지 검사하고, 새로운 edge(tuple)가 발견되면 2, 기존 edge의 hit-count bucket만 달라졌으면 1, 변화가 없으면 0을 반환한다.

정리하면,
| 반환값 | 의미                        | AFL의 해석                    |
| ------ | --------------------------- | ----------------------------- |
| `0`    | 새로운 정보 없음            | 버릴 가능성이 높음            |
| `1`    | hit count bucket 변화       | 약간 흥미로운 입력            |
| `2`    | 새로운 tuple 발견           | queue에 저장할 가치가 큼      |

또 하나 중요한 점은 AFL이 단순히 hit count를 정확한 숫자로만 보지 않는다는 것이다. 
AFL은 edge hit count를 여러 bucket으로 정규화해 비교한다.
```
1
2
3
4~7
8~15
16~31
32~127
128+
```

AFL 문서는 hit count를 위와 같은 bucket으로 나누며, 같은 bucket 안의 변화는 무시하고 다른 bucket으로 넘어가는 변화는 흥미로운 control-flow 변화로 본다고 설명한다.

이 방식의 장점은 단순하다. 루프를 100번 돌았는지 101번 돌았는지까지 세밀하게 구분하면 noise가 너무 많다. 하지만 루프를 1번 돌던 입력이 갑자기 32번 이상 돌게 된다면, 그건 프로그램 동작이 의미 있게 달라진 것일 가능성이 높다.

따라서 AFL의 coverage 판단은 단순한 “방문 여부”가 아니라 다음 두 가지를 함께 본다.
* edge tuple이 새로 등장했는가?
* edge의 hit count bucket이 달라졌는가?

따라서 Instrumentation은 coverage를 **수집하기 위한 수단**이고, Coverage는 **AFL이 새로운 실행 경로를 판단하는 기준**이라고 볼 수 있다.

## 3. Fork Server — AFL은 어떻게 빠르게 실행하는가?

Fuzzing은 같은 프로그램을 수천, 수만, 수백만 번 반복 실행한다.
만약 매 입력마다 다음 과정을 전부 반복하면 비용이 커진다.

```
fork()
execve()
dynamic linking
libc initialization
program initialization
main 진입
```
AFL은 이 오버헤드를 줄이기 위해 fork server를 사용한다.

Fork server의 핵심 아이디어는 다음과 같다.
* 타겟 프로그램을 처음 한 번만 execve()로 실행한다.
* 초기화가 끝난 지점에서 멈춘다.
* 이후 입력마다 새로 execve()하지 않고 fork()만 수행한다.

AFL technical details 문서에서도 fork server는 fuzzed process가 `execve()`, linking, libc initialization을 한 번만 거치고, 이후 copy-on-write를 활용해 stopped process image에서 clone된다고 설명한다.

AFL 원본에서는 fork server가 instrumentation payload의 일부로 target binary 안에 들어간다. `config.h`에서는 fork server 통신에 사용하는 file descriptor를 `FORKSRV_FD = 198`로 정의한다. 따라서 AFL은 보통 fd 198, 199를 사용해 fuzzer와 target 내부 fork server가 통신한다.

흐름을 단순화하면 다음과 같다.
```
[afl-fuzz]                      [target 내부 fork server]

pipe 생성
fork()
execve(target)
                                초기화 완료
                                fork server 준비 신호 전송
준비 신호 수신
테스트 입력 준비
실행 명령 전송  ───────────────▶ fork()
                                child 실행
                                child 종료 대기
상태 수신 ◀──────────────────── 종료 status 전달
다음 입력 준비
반복

```
Fork server가 없으면 매번 `execve()`를 반복해야 한다. Fork server가 있으면 target을 한 번만 로드한 뒤, 이후에는 `fork()` 중심으로 반복 실행할 수 있다. 그래서 AFL은 빠른 target에서 큰 성능 이득을 얻을 수 있다. AFL 문서에서는 fast target에서 fork server가 보통 1.5배에서 2배 정도의 성능 이득을 줄 수 있다고 설명한다.

즉 Fork Server는 AFL의 *“속도 최적화 장치”*이다.
Coverage가 좋은 입력을 찾는 논리라면, Fork Server는 그 논리를 *매우 많이 반복할 수 있게 해주는 실행 구조*이다.

## 4. Mutation — AFL은 입력을 어떻게 변형하는가?

Mutation은 AFL이 새로운 경로를 찾기 위해 seed input을 변형하는 과정이다.

AFL은 초기 입력을 queue에 넣고, queue에서 하나씩 꺼내 변형한다. 변형된 입력이 새로운 edge나 hit count 변화를 만들면 다시 queue에 추가한다. README의 알고리즘 설명에서도 AFL은 초기 test case를 queue에 넣고, test case를 trim한 뒤, 다양한 fuzzing 전략으로 반복 mutation을 수행하며, instrumentation이 새로운 state transition을 기록하면 그 입력을 queue에 추가한다고 설명한다.

AFL의 mutation은 크게 두 단계로 볼 수 있다.

1. Deterministic stage
   → 정해진 방식으로 순차적으로 바꿔본다.

2. Non-deterministic stage
   → 랜덤성이 강한 havoc, splice 등을 수행한다.

AFL technical details 문서에 따르면 초기에는 deterministic 전략이 많이 사용된다. 여기에는 sequential bit flips, 작은 정수의 addition/subtraction, 0, 1, INT_MAX 같은 interesting integer 삽입이 포함된다. 이후에는 stacked bit flips, insertion, deletion, arithmetic, splicing 같은 비결정적 변이가 수행된다.

`afl-fuzz.c`에도 fuzzing stage enum이 정의되어 있다. 여기에는 STAGE_FLIP1, STAGE_FLIP2,...,STAGE_SPLICE 등이 포함된다.

각 mutation 단계의 의미는 다음처럼 정리할 수 있다.

```
bit flip
  → 입력의 특정 bit를 뒤집는다.

arithmetic
  → byte, word, dword 단위로 작은 값을 더하거나 뺀다.

interesting value
  → 0, -1, 127, 128, 255, INT_MAX 같은 경계값을 넣는다.

dictionary extras
  → 사용자가 준 token이나 자동 탐지된 token을 삽입/치환한다.

havoc
  → 여러 변이를 랜덤하게 쌓아서 적용한다.

splice
  → 서로 다른 queue input을 잘라 붙여 새로운 입력을 만든다.
```
여기서 중요한 점은 AFL의 mutation은 **프로그램 구조를 완전히 이해한 상태에서 수행되는 것은 아니라는 점**이다. AFL은 대부분의 mutation을 format-agnostic하게 수행한다. 대신 instrumentation과 coverage feedback이 “이 변이가 의미 있었는가?”를 판단해준다.

즉 AFL의 강점은 mutation 하나하나가 똑똑해서가 아니라, 다음 feedback loop가 빠르게 반복된다는 데 있다.
```
입력 변형
    ↓
실행
    ↓
coverage 확인
    ↓
새로운 동작이면 queue 저장
    ↓
다시 변형
```
그래서 AFL은 간단한 mutation 전략으로도 점점 더 깊은 경로를 탐색할 수 있다.

### 실습 4
Mutation 실습에서는 AFL이 seed input을 queue에 저장하고,
이를 기반으로 새로운 입력을 생성하는 구조를 확인했다.

<img src="assets/img/AFL/afl-mut.png" width="400" alt="AFL 실습 이미지">

<img src="assets/img/AFL/afl-mut2.png" width="400" alt="AFL 실습 이미지">

실습 결과 out/queue에는 id:000000,orig:seed.txt 파일이 존재했다.
이 파일은 사용자가 입력 디렉터리에 넣어둔 초기 seed가 AFL의 queue에 등록된 것이다.

반면 out/crashes 디렉터리는 비어 있었다.
이는 현재 실습 target이 crash를 유발하는 코드가 없거나,
짧은 실행 시간 동안 crash 입력을 발견하지 못했기 때문이다.

AFL은 queue에 있는 입력을 bit flip, arithmetic, interesting value, havoc 등의 방식으로 변형한다.
변형된 입력이 새로운 coverage를 만들면 out/queue에 새로운 id 파일로 저장되고,
crash를 일으키면 out/crashes에 저장된다.

지금 queue가 seed 하나만 있는 이유는 테스트 코드가 단순하기 때문입니다.
여러 seed를 넣으면 queue의 초기 seed가 여러 개 들어가서 AFL이 다양한 실행 경로를 기준으로 mutation을 시작할 수 있는데,
hello 하나만 있을 때는 AFL이 우연히 "AFL MAGIC" 문자열을 만들어야 deep branch에 도달하지만,
처음부터 AFL, AFL MAGIC 같은 seed를 함께 제공하면 더 많은 branch를 출발점으로 삼을 수 있습니다.
따라서 seed corpus가 다양할수록 AFL이 새로운 coverage를 찾을 가능성도 높아지는걸 알 수 있습니다.

## 5. 흐름정리

AFL의 네 가지 핵심 요소는 따로 움직이는 것이 아니라 하나의 loop를 구성한다.

```
Instrumentation
  target program에 edge 기록 코드를 삽입한다.

Coverage
  shared memory bitmap을 보고 새로운 edge인지 판단한다.

Fork Server
  target을 빠르게 반복 실행한다.

Mutation
  기존 입력을 바꿔 새로운 coverage를 만들려고 시도한다.
```
하나의 입력이 처리되는 과정을 전체적으로 보면 다음과 같다.

```
queue에서 seed 선택
        ↓
mutation으로 새 입력 생성
        ↓
fork server가 target child 실행
        ↓
target 내부 instrumentation 코드 실행
        ↓
shared memory bitmap 업데이트
        ↓
afl-fuzz가 has_new_bits()로 분석
        ↓
새로운 tuple이면 queue에 추가
        ↓
반복
```

따라서 AFL은 단순한 랜덤 입력 생성기가 아니라, **instrumentation으로 관찰**하고,  
** coverage로 판단**하고, **fork server로 빠르게 반복**하고, **mutation으로  
새로운 입력을 생성하는 coverage-guided fuzzer**라고 정리할 수 있다.
