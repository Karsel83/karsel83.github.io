## 1. 정의 (What is Instrumentation?)
**Instrumentation(계측)**이란 프로그램의 실행 흐름을 관찰하기 위해 바이너리에 추가적인 코드를 삽입하는 기술입니다.  
AFL에서의 계측은 퍼저가 "어떤 입력값이 새로운 경로를 실행시켰는가?"를 판단할 수 있게 하는 '눈'의 역할을 합니다.

---

## 2. 계측 메커니즘 (Mechanism)
AFL은 효율성을 위해 컴파일 타임(Compile-time)에 어셈블리 코드를 수정하는 방식을 채택합니다.

### 2-1. 컴파일 파이프라인 가로채기
이미지 **AFL-BUILD1.png**에서 수행한 `afl-gcc` 컴파일 과정의 핵심은 다음과 같습니다.
*   **Wrapper**: `afl-gcc`는 실제 컴파일러(`gcc`)를 감싸는 래퍼입니다.
*   **Assembler Substitution**: `gcc`가 소스코드를 어셈블리(`.s`)로 변환하면, AFL은 시스템 기본 어셈블러 대신 `afl-as`를 호출하여 코드를 삽입합니다.

### 2-2. 삽입 지점 (Instrumentation Points)
`afl-as`는 모든 명령어가 아닌, 실행 흐름이 변하는 **기본 블록(Basic Block)**의 시작 지점에 **Trampoline**이라 불리는 코드를 삽입합니다.
*   조건부 분기문 (`if`, `switch` 등)
*   함수의 시작점
*   점프 라벨 이후의 첫 명령어

---

## 3. 구현 원리 (Implementation Details)

AFL Instrumentation의 정수는 **'최소한의 오버헤드로 최대한의 경로 정보를 얻는 것'**에 있습니다. 이를 위해 다음과 같은 알고리즘과 데이터 구조를 사용합니다.

### 3-1. 비트맵과 공유 메모리 (Shared Memory)
AFL은 퍼저와 타겟 바이너리 간의 통신을 위해 64KB 크기의 공유 메모리(`trace_bits`)를 생성합니다.
*   **MAP_SIZE (64KB)**: 각 분기점은 이 비트맵의 특정 인덱스에 접근하여 실행 횟수를 카운트합니다.
*   **Bitmap Update**: 타겟이 실행될 때마다 `trace_bits[location]++` 연산을 수행하여 어떤 경로를 밟았는지 기록합니다.

### 3-2. 방향성 보존 알고리즘 (Edge Coverage)
단순히 "어느 블록을 방문했는가"만 기록하면 $A \to B$와 $B \to A$의 차이를 알 수 없습니다. AFL은 이를 해결하기 위해 **Shift-XOR** 기법을 사용합니다.

```assembly
/* 실제 __afl_maybe_log 내부의 로직 원리 */
cur_location = <COMPILE_TIME_RANDOM_ID>;          // 현재 블록의 고유 ID (rcx)
index = cur_location ^ prev_location;            // 이전 위치와 현재 위치를 XOR하여 인덱스 생성
trace_bits[index]++;                             // 해당 경로(Edge) 방문 횟수 증가
prev_location = cur_location >> 1;               // 현재 위치를 1비트 밀어서 '이전 위치'로 저장
```

>> 1 (Right Shift): 현재 위치를 오른쪽으로 1비트 밀어줌으로써, $A \to B$($A_{shifted} \oplus B$)와 $B \to A$($B_{shifted} \oplus A$)의 결과값을 다르게 만듭니다.
이를 통해 경로의 방향성을 완벽하게 보존합니다.
### 3-3. Context Saving (레지스터 보호)
계측 코드는 프로그램 본래의 동작에 영향을 주면 안 됩니다. 따라서 afl-as.h의 Trampoline 코드는 실행 직후 다음을 수행합니다.
* Stack Protection: x86-64의 Red Zone을 고려하여 스택 포인터를 안전하게 조정합니다.
* Register Saving: rax, rcx, rdx 및 EFLAGS 등 연산에 사용될 레지스터의 원래 값을 보관했다가 계측이 끝나면 복구합니다.

## 4. 실습 결과 분석 (Based on Lab Data)  

### 4-1. 커버리지 기반 탐색  
(AFL-BUILD2.png)
실습 화면인 AFL-BUILD2.png를 통해 Instrumentation의 실효성을 확인할 수 있습니다.  
* Map Density: 0.01% / 0.02% 수치는 계측 코드가 심어진 특정 분기들을 입력값이 정확히 통과하고 있음을 나타냅니다.  
* New Path: total paths : 4 결과는 vuln.c 내부의 if 조건문을 통과할 때마다 새로운 경로로 인식하여 퍼저가 시드(Seed)를 확장했음을 증명합니다.

### 4-2. 크래시 탐지 및 디버깅 (AFL-GDB.png)  
* 학습 능력: 퍼저는 계측 정보를 통해 특정 문자열(A, AB 등)이 더 깊은 코드를 실행시킨다는 것을 학습했습니다.  
* Crash Analysis: 최종적으로 strcpy의 취약점을 트리거하는 입력을 생성했으며, AFL-GDB.png 화면처럼 __strcpy_chk에서 Stack Overflow가 발생했음을 확인했습니다.

## 5. 결론  
AFL의 Instrumentation은 어셈블리 레벨의 코드 삽입과 공유 메모리 비트맵을 통해 최소한의 오버헤드로 최대한의 경로 정보를 수집합니다. 이번 실습을 통해 계측 코드가 어떻게 퍼저의 지능적인 탐색을 돕고, 최종적으로 메모리 오염 취약점을 찾아내는지 그 전 과정을 검증하였습니다.
