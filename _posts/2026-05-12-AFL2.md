## 0. 목표

이 글의 목표는 **AFL 코드**를 직접 보면서 AFL의 핵심 메커니즘을 이해하는 것이다.  

AFL은 단순히 랜덤 입력을 계속 넣는 퍼저가 아니라, target program의 실행 경로를 관찰하고 새로운 coverage를 만드는 입력을 보존하는 **coverage-guided fuzzer**이다.

이번 글에서는 AFL을 다음 네 가지 관점에서 분석한다.

```text
1. Instrumentation
   → target program에 coverage 기록 코드를 삽입하는 과정

2. Coverage
   → 실행 결과 bitmap을 비교해 새로운 경로인지 판단하는 과정

3. Fork Server
   → target을 빠르게 반복 실행하기 위한 구조

4. Mutation
   → seed input을 변형해 새로운 경로를 찾는 과정
```

글의 구성은 단순 실습 리포트가 아니라, **코드 로직을 먼저 분석하고 VM 실습으로 확인하는 방식**으로 진행한다.

```text
관련 코드 확인
        ↓
핵심 함수와 변수 분석
        ↓
코드 흐름 해석
        ↓
VM 실습 결과로 검증
```

---

## 1. Instrumentation 코드 분석

AFL의 Instrumentation은 target program의 실행 흐름을 추적하기 위해 컴파일 과정에서 coverage 기록 코드를 삽입하는 기능이다.

Instrumentation 관련 핵심 파일은 다음과 같다.

```text
afl-gcc.c  : gcc/clang wrapper
afl-as.c   : assembly 파싱 및 instrumentation 삽입
afl-as.h   : 실제 삽입되는 trampoline과 __afl_maybe_log payload
```

전체 흐름은 다음과 같다.

```text
afl-gcc
    ↓
gcc 실행 인자 수정
    ↓
gcc가 assembly 생성
    ↓
afl-as가 assembly를 가로채서 분석
    ↓
branch 주변에 trampoline 삽입
    ↓
instrumented binary 생성
```

즉 AFL은 source code를 직접 수정하지 않고, **컴파일 중간에 생성된 assembly를 수정하는 방식**으로 instrumentation을 수행한다.

---

### 1-1. `afl-gcc.c` — gcc wrapper와 assembler 교체

먼저 `afl-gcc.c`에서 instrumentation 흐름의 시작점을 확인했다.

<img src="assets/img/AFL/add1.png" width="400" alt="AFL 실습 이미지">

`grep` 결과 `edit_params()` 함수와 `-B`, `-no-integrated-as` 옵션이 확인된다.

```text
118: static void edit_params(u32 argc, char** argv)
217: cc_params[cc_par_cnt++] = "-B";
221: cc_params[cc_par_cnt++] = "-no-integrated-as";
```

`edit_params()`는 사용자가 넘긴 컴파일 인자를 그대로 gcc에 전달하지 않고, AFL이 필요한 옵션을 추가하거나 기존 옵션을 제거하는 함수이다.

<img src="assets/img/AFL/add3.png" width="400" alt="AFL 실습 이미지">

함수 내부에서는 먼저 실제로 호출할 컴파일러를 결정한다.

<img src="assets/img/AFL/add4.png" width="400" alt="AFL 실습 이미지">

내 실습에서는 다음 명령어를 사용했다.

```bash
./afl-gcc test.c -o test_afl
```

따라서 기본적으로 `gcc`가 선택된다.  
즉 `afl-gcc`는 직접 컴파일을 수행하는 프로그램이 아니라, 실제 gcc를 실행하기 위한 인자 배열을 구성하는 wrapper이다.

---

### 1-2. 기존 옵션 제거와 `-B as_path` 추가

`edit_params()`는 사용자가 이미 넘긴 일부 옵션을 제거한다.

[add5]

특히 기존 `-B` 옵션은 무시된다.  
`-B`는 gcc가 assembler 같은 toolchain 구성 요소를 찾는 경로를 지정하는 옵션이다.

AFL은 반드시 자신이 지정한 `afl-as`가 사용되도록 해야 하므로 기존 `-B`를 제거하고, 뒤에서 AFL의 경로를 다시 넣는다.  
또한 `-integrated-as`와 `-pipe`도 제거된다. clang의 내장 assembler나 pipe 방식이 사용되면 `afl-as`가 중간 assembly를 수정하기 어렵기 때문이다.

가장 중요한 코드는 다음 두 줄이다.

[add6]

```c
cc_params[cc_par_cnt++] = "-B";
cc_params[cc_par_cnt++] = as_path;
```

이 코드는 gcc가 assembler를 찾을 때 `as_path`를 먼저 보도록 만든다.

```text
일반 gcc 빌드:
gcc → as → object file

AFL 빌드:
afl-gcc → gcc -B <AFL path> → afl-as → 실제 as → object file
```

따라서 `afl-gcc.c`의 역할은 instrumentation을 직접 삽입하는 것이 아니라, `afl-as`가 assembly를 수정할 수 있도록 컴파일 파이프라인을 우회시키는 것이다.

---

### 1-3. `afl-as.c` — assembly 파싱과 trampoline 삽입

실제 instrumentation 삽입은 `afl-as.c`에서 수행된다.

<img src="assets/img/AFL/add2.png" width="400" alt="AFL 실습 이미지">

`grep` 결과를 보면 `add_instrumentation()` 함수와 `"Instrumented %u locations"` 출력 문자열을 확인할 수 있다.

```text
227: static void add_instrumentation(void)
464: Instrumented %u locations
```

`add_instrumentation()`은 gcc가 생성한 assembly 파일을 읽으면서 계측을 넣을 위치를 찾는다.  
AFL은 모든 줄에 코드를 넣는 것이 아니라, branch label이나 conditional branch처럼 실행 흐름이 갈라지는 지점을 중심으로 trampoline을 삽입한다.

대표적인 계측 대상은 다음과 같다.

```text
main:       → function entry
.L0:        → GCC branch label
.LBB0_0:    → clang branch label
jnz foo     → conditional branch
```

반대로 `.data`, `.bss`, inline assembly, `jmp` 같은 unconditional jump는 계측 대상에서 제외된다.  
즉 `afl-as.c`는 assembly를 파싱하면서 실행 경로 추적에 의미 있는 지점만 골라 instrumentation을 삽입한다.

---

### 1-4. `afl-as.h` — trampoline과 `__afl_maybe_log`

`afl-as.c`가 “어디에 넣을지”를 결정한다면, `afl-as.h`는 “무엇을 넣을지”를 정의한다.

내 실습 환경은 64-bit였기 때문에 `AFL TRAMPOLINE (64-BIT)` 코드가 사용된다.

```text
141: AFL TRAMPOLINE (64-BIT)
150: call __afl_maybe_log
399: __afl_maybe_log:
```

trampoline의 역할은 다음과 같다.

```text
1. 일부 register를 저장한다.
2. 현재 위치 ID를 register에 넣는다.
3. __afl_maybe_log를 호출한다.
4. register를 복원한다.
5. 원래 프로그램 흐름으로 돌아간다.
```

`__afl_maybe_log`는 실제 coverage bitmap을 갱신한다.  
개념적으로는 다음과 같은 동작을 수행한다.

```c
cur_location = <COMPILE_TIME_RANDOM>;
shared_mem[cur_location ^ prev_location]++;
prev_location = cur_location >> 1;
```

여기서 핵심은 `cur_location ^ prev_location`이다.  
AFL은 단순히 현재 basic block만 기록하지 않고, 이전 위치에서 현재 위치로 이동한 **edge**를 bitmap index로 기록한다.

또한 `prev_location = cur_location >> 1`을 사용하는 이유는 XOR의 대칭성 때문이다.

```text
A ^ B == B ^ A
```

그냥 XOR만 사용하면 `A → B`와 `B → A`가 같은 값이 될 수 있다.  
AFL은 현재 위치를 1비트 shift해서 저장함으로써 edge의 방향성과 loop를 더 잘 구분한다.

---

### 1-5. 실습으로 확인한 Instrumentation 동작

분석에 사용한 target 코드는 다음과 같다.

<img src="assets/img/AFL/AFL1-TEST.png" width="400" alt="AFL 실습 이미지">

이 코드는 입력값에 따라 `normal branch`, `first branch`, `deep branch`로 실행 경로가 달라진다.

일반 gcc와 AFL의 `afl-gcc`로 각각 컴파일했다.

<img src="assets/img/AFL/AFL1.png" width="400" alt="AFL 실습 이미지">

```bash
gcc test.c -o test_gcc
./afl-gcc test.c -o test_afl
```

AFL 빌드 결과 다음 메시지가 출력되었다.

```text
[+] Instrumented 9 locations (64-bit, non-hardened mode, ratio 100%).
```

이는 `afl-as.c`의 `add_instrumentation()`이 assembly를 분석한 결과, 총 9개의 위치에 trampoline을 삽입했다는 의미이다.

이후 `AFL_KEEP_ASSEMBLY=1`로 instrumented assembly를 보존하고 확인했다.

<img src="assets/img/AFL/AFL-TRAMPOLINE.png" width="400" alt="AFL 실습 이미지">

```bash
grep -n "AFL TRAMPOLINE" assembly_here/.afl-*.s
grep -n "__afl_maybe_log" assembly_here/.afl-*.s
```

결과적으로 `AFL TRAMPOLINE`이 9개 확인되었고, 각 trampoline에서 `__afl_maybe_log` 호출도 확인되었다.  
이는 앞에서 출력된 `Instrumented 9 locations`와 정확히 대응된다.

정리하면 AFL instrumentation 흐름은 다음과 같다.

```text
afl-gcc.c
  → -B as_path로 afl-as 연결

afl-as.c
  → assembly 분석 후 trampoline 삽입

afl-as.h
  → trampoline과 __afl_maybe_log 정의

__afl_maybe_log
  → coverage bitmap 갱신
```

## 2. Coverage 코드 분석

Instrumentation이 target binary 안에 coverage 기록 코드를 삽입한다면, Coverage는 그 기록을 AFL이 어떻게 해석하는지에 해당한다.

Coverage 관련 핵심 파일은 다음과 같다.

```text
config.h    : MAP_SIZE, SHM_ENV_VAR 정의
afl-as.h    : trace bitmap 갱신 payload
afl-fuzz.c  : has_new_bits()로 새로운 coverage 판단
```

---

### 2-1. Coverage bitmap 구조

AFL은 실행 결과를 64KB shared memory bitmap에 기록한다.

`config.h`에서는 bitmap 크기를 다음과 같이 정의한다.

```c
#define MAP_SIZE_POW2 16
#define MAP_SIZE (1 << MAP_SIZE_POW2)
```

따라서 bitmap 크기는 다음과 같다.

```text
2^16 = 65536 bytes = 64KB
```

target program은 instrumentation 코드 실행 중 이 bitmap을 갱신하고, `afl-fuzz`는 실행이 끝난 뒤 bitmap을 확인한다.

```text
target 실행
    ↓
__afl_maybe_log가 bitmap 갱신
    ↓
afl-fuzz가 trace_bits 확인
    ↓
새로운 coverage 여부 판단
```

---

### 2-2. `has_new_bits()` — 새로운 coverage 판단

Coverage 판단의 핵심 함수는 `afl-fuzz.c`의 `has_new_bits()`이다.

개념적으로 AFL은 다음 두 bitmap을 비교한다.

```text
trace_bits   : 이번 실행에서 target이 기록한 coverage
virgin_bits  : 아직 발견되지 않은 coverage 정보
```

`trace_bits`에 값이 있고, 그 위치가 `virgin_bits`에도 남아 있다면 AFL은 이번 입력이 새로운 정보를 만들었다고 판단한다.

`has_new_bits()`의 반환값은 다음처럼 해석할 수 있다.

| 반환값 | 의미 | 해석 |
|---|---|---|
| `0` | 새로운 정보 없음 | 버릴 가능성이 높음 |
| `1` | hit count 변화 | 약간 흥미로운 입력 |
| `2` | 새로운 tuple 발견 | queue에 저장할 가치가 큼 |

AFL은 단순히 “방문했는가”만 보는 것이 아니라, edge hit count 변화도 bucket 단위로 판단한다.

```text
1
2
3
4~7
8~15
16~31
32~127
128+
```

이 방식은 루프 횟수의 아주 작은 차이에는 민감하게 반응하지 않으면서, 의미 있는 실행 흐름 변화는 잡아낼 수 있게 한다.

---

### 2-3. 실습으로 확인한 Coverage 차이

같은 target binary에 서로 다른 입력을 넣고 `afl-showmap`으로 coverage tuple 수를 확인했다.

먼저 normal 입력이다.

<img src="assets/img/AFL/afl2-normal.png" width="400" alt="AFL 실습 이미지">

```text
normal branch
Captured 5 tuples
```

다음은 `AFL` 입력이다.

<img src="assets/img/AFL/afl2-afl.png" width="400" alt="AFL 실습 이미지">

```text
first branch
Captured 6 tuples
```

마지막은 `AFL MAGIC` 입력이다.

<img src="assets/img/AFL/afl2-deep.png" width="400" alt="AFL 실습 이미지">

```text
first branch
deep branch
Captured 7 tuples
```

결과를 정리하면 다음과 같다.

<img src="assets/img/AFL/add7.png" width="400" alt="AFL 실습 이미지">

```text
hello       → normal branch              → 5 tuples
AFL         → first branch               → 6 tuples
AFL MAGIC   → first branch + deep branch → 7 tuples
```

즉 입력값이 더 깊은 branch에 도달할수록 AFL이 기록한 tuple 수가 증가했다.  
이는 instrumentation이 실제 실행 경로 차이를 bitmap에 기록하고 있음을 보여준다.

AFL은 fuzzing 중 mutation으로 생성한 입력이 이런 새로운 tuple을 만들면, 해당 입력을 흥미로운 입력으로 판단하고 queue에 저장한다.

## 3. Fork Server 코드 분석

AFL은 target program을 매우 많이 반복 실행해야 한다.  
만약 매 입력마다 target을 새로 `execve()`하면 dynamic linking, libc 초기화, program 초기화 비용이 계속 발생한다.

AFL은 이 오버헤드를 줄이기 위해 **Fork Server**를 사용한다.  
Fork Server는 target을 처음 한 번만 실행한 뒤, 이후 testcase마다 `fork()`만 수행해 빠르게 반복 실행하는 구조이다.

Fork Server 관련 핵심 파일은 다음과 같다.

```text
config.h    : Fork Server 통신 FD 정의
afl-as.h    : target binary에 삽입되는 fork server payload
afl-fuzz.c  : fork server 시작 및 child process 관리
```

---

### 3-1. `config.h` — Fork Server 통신 FD 확인

먼저 `config.h`에서 Fork Server가 사용하는 file descriptor를 확인했다.

```bash
grep -n "FORKSRV_FD" config.h
```

결과는 다음과 같았다.

<img src="assets/img/AFL/fork1.png" width="400" alt="AFL 실습 이미지">

`FORKSRV_FD`는 `afl-fuzz`와 target 내부 fork server가 통신할 때 사용하는 기준 file descriptor이다.  
AFL은 `FORKSRV_FD`와 `FORKSRV_FD + 1`, 즉 fd 198과 199를 사용한다.

```text
fd 198 : afl-fuzz → fork server 실행 명령 전달
fd 199 : fork server → afl-fuzz 실행 상태 전달
```

즉 Fork Server는 단순히 target 안에서 `fork()`만 수행하는 코드가 아니라, `afl-fuzz`와 pipe를 통해 명령과 결과를 주고받는 구조이다.

---

### 3-2. `afl-as.h` — target 내부 Fork Server payload

다음으로 `afl-as.h`에서 `__afl_forkserver`를 확인했다.

```bash
grep -n "__afl_forkserver" afl-as.h
```

결과는 다음과 같았다.

<img src="assets/img/AFL/fork2.png" width="400" alt="AFL 실습 이미지">

두 위치가 나오는 이유는 `afl-as.h`에 32-bit payload와 64-bit payload가 모두 정의되어 있기 때문이다.  
내 실습 환경은 64-bit였으므로 뒤쪽 64-bit fork server payload를 기준으로 분석했다.

<img src="assets/img/AFL/fork3-1.png" width="400" alt="AFL 실습 이미지">

target 내부 fork server가 afl-fuzz에게 “fork server 준비 완료” 신호를 보내는 과정이다.
이 신호를 받은 afl-fuzz는 fork server가 정상적으로 올라왔다고 판단한다.

그 다음 fork server는 __afl_fork_wait_loop로 들어간다.

<img src="assets/img/AFL/fork3-2.png" width="400" alt="AFL 실습 이미지">

즉 fork server는 afl-fuzz가 다음 testcase를 실행하라고 명령할 때까지 기다린다.
명령을 받으면 fork server는 fork()를 호출해 child process를 만든다.

<img src="assets/img/AFL/fork3-3.png" width="400" alt="AFL 실습 이미지">

여기서 fork()의 반환값이 0이면 child process이므로 __afl_fork_resume으로 이동해 원래 target program 실행을 계속한다.
반대로 반환값이 0보다 크면 parent, 즉 fork server이므로 child PID를 저장하고 afl-fuzz에게 전달한다.

<img src="assets/img/AFL/fork3-4.png" width="400" alt="AFL 실습 이미지">

이후 parent fork server는 child가 끝날 때까지 기다린다.

<img src="assets/img/AFL/fork3-5.png" width="400" alt="AFL 실습 이미지">

child가 종료되면 fork server는 종료 status를 다시 afl-fuzz에게 전달한다.

<img src="assets/img/AFL/fork3-6.png" width="400" alt="AFL 실습 이미지">

다시 __afl_fork_wait_loop로 돌아가 다음 실행 명령을 기다린다.
마지막으로 child process는 __afl_fork_resume 경로로 이동한다.

<img src="assets/img/AFL/fork3-7.png" width="400" alt="AFL 실습 이미지">

child는 fork server 통신에 쓰던 fd 198, 199를 닫고 원래 target program 흐름으로 복귀한다.
즉 실제 testcase 실행은 child process가 담당하고, parent fork server는 child의 PID와 종료 status를 afl-fuzz에게 전달하는 관리자 역할을 한다.

정리하면 __afl_forkserver의 핵심 흐름은 다음과 같다.

```
1. write(FORKSRV_FD + 1)
   → fork server 준비 완료 신호 전송

2. read(FORKSRV_FD)
   → afl-fuzz의 실행 명령 대기

3. fork()
   → testcase 실행용 child process 생성

4. write(FORKSRV_FD + 1)
   → child PID를 afl-fuzz에게 전달

5. waitpid(child_pid)
   → child 종료 대기

6. write(FORKSRV_FD + 1)
   → child 종료 status 전달

7. jmp __afl_fork_wait_loop
   → 다음 testcase 실행 명령 대기
```
이 구조 덕분에 AFL은 매 testcase마다 target을 새로 execve()하지 않고, 이미 초기화된 target process image에서 fork()만 반복할 수 있다.

---

### 3-3. `afl-fuzz.c` — fuzzer 쪽 Fork Server 관리

`afl-fuzz.c`에서도 fork server와 관련된 핵심 함수와 변수를 확인했다.

```bash
grep -n "init_forkserver" afl-fuzz.c
grep -n "run_target" afl-fuzz.c
grep -n "fork server is up" afl-fuzz.c
grep -n "forksrv_pid" afl-fuzz.c | head
grep -n "child_pid" afl-fuzz.c | head
```

확인 결과 `init_forkserver()`와 `run_target()`이 fork server 실행 흐름의 핵심임을 알 수 있었다.

<img src="assets/img/AFL/fork3.png" width="400" alt="AFL 실습 이미지">

```text
init_forkserver() : fork server 초기화
run_target()      : testcase 실행 요청 및 결과 수신
forksrv_pid       : fork server process PID
child_pid         : 실제 testcase를 실행하는 child process PID
```

특히 `forksrv_pid`와 `child_pid`가 분리되어 있다는 점이 중요하다.

```text
forksrv_pid
  → target 내부에서 계속 살아 있는 fork server

child_pid
  → 매 testcase 실행을 위해 fork된 자식 프로세스
```

즉 AFL은 target을 매번 새로 실행하는 것이 아니라, 살아 있는 fork server에게 실행 명령을 보내고, fork server가 child를 만들어 testcase를 실행하게 한다.

---

### 3-4. 실습으로 확인한 Fork Server 동작

<img src="assets/img/AFL/afl3-fork.png" width="400" alt="AFL 실습 이미지">

실제로 `afl-fuzz`를 실행했을 때 다음 메시지가 출력되었다.

```text
All right - fork server is up.
```

이 메시지는 `afl-as.h`에 정의된 `__afl_forkserver` payload가 target binary 안에서 정상적으로 실행되었고, `afl-fuzz`와 초기 handshake에 성공했다는 의미이다.

정리하면 Fork Server 흐름은 다음과 같다.

```text
afl-fuzz
    ↓
target 최초 실행
    ↓
target 내부 __afl_forkserver 실행
    ↓
준비 완료 신호 write()
    ↓
afl-fuzz가 실행 명령 전달
    ↓
fork server가 fork()
    ↓
child가 testcase 실행
    ↓
종료 status 전달
```

따라서 Fork Server는 AFL이 많은 testcase를 빠르게 실행할 수 있게 해주는 실행 최적화 구조라고 볼 수 있다.

---

## 4. Mutation 코드 분석

Mutation은 AFL이 기존 seed input을 변형해 새로운 coverage를 찾는 과정이다.  
AFL의 mutation 로직은 대부분 `afl-fuzz.c` 안에서 확인할 수 있다.

Mutation 관련 핵심 요소는 다음과 같다.

```text
queue_entry           : testcase 정보를 저장하는 구조체
add_to_queue()        : 새로운 testcase를 queue에 추가
save_if_interesting() : 실행 결과가 의미 있는지 판단하고 저장
fuzzing stage enum    : bit flip, arithmetic, havoc, splice 등 mutation 단계
```

---

### 4-1. Mutation 관련 코드 위치 확인



먼저 `afl-fuzz.c`에서 mutation과 관련된 구조체와 함수를 검색했다.

```bash
grep -n "struct queue_entry" afl-fuzz.c
grep -n "add_to_queue" afl-fuzz.c
grep -n "save_if_interesting" afl-fuzz.c
grep -n "STAGE_FLIP" afl-fuzz.c
grep -n "STAGE_HAVOC" afl-fuzz.c
grep -n "STAGE_SPLICE" afl-fuzz.c
```

결과를 보면 queue 관리와 mutation stage가 모두 `afl-fuzz.c`에 구현되어 있음을 확인할 수 있다.

<img src="assets/img/AFL/mut1.png" width="400" alt="AFL 실습 이미지">

```text
struct queue_entry       → testcase 메타데이터 구조체
add_to_queue()           → queue에 새 testcase 추가
save_if_interesting()    → 의미 있는 입력인지 판단 후 저장
STAGE_FLIP               → bit flip 계열 mutation
STAGE_HAVOC              → 랜덤 조합 mutation
STAGE_SPLICE             → 서로 다른 입력을 잘라 붙이는 mutation
```

즉 Mutation은 단순히 입력을 무작위로 바꾸는 기능 하나가 아니라, queue 관리와 coverage 판단 로직과 함께 동작한다.

---

### 4-2. `queue_entry` — testcase 관리 구조체


`queue_entry`는 AFL queue에 저장되는 하나의 testcase를 표현한다.

<img src="assets/img/AFL/mut2.png" width="400" alt="AFL 실습 이미지">

주요 필드는 다음처럼 해석할 수 있다.

```text
fname        : queue에 저장된 testcase 파일 이름
len          : 입력 길이
was_fuzzed   : 이미 fuzzing이 수행되었는지 여부
passed_det   : deterministic stage 통과 여부
has_new_cov  : 새로운 coverage를 만든 입력인지 여부
bitmap_size  : coverage bitmap 크기
exec_us      : 실행 시간
depth        : testcase 파생 깊이
next         : 다음 queue entry
```

즉 AFL의 queue는 단순한 파일 목록이 아니라, 각 testcase의 실행 결과와 fuzzing 상태를 함께 저장하는 linked list 구조이다.

---

### 4-3. Mutation stage enum

`queue_entry` 바로 아래에는 AFL의 mutation stage enum도 정의되어 있다.

<img src="assets/img/AFL/mut3.png" width="400" alt="AFL 실습 이미지">

각 stage는 다음과 같은 의미를 가진다.

```text
STAGE_FLIP
  → bit 단위 flip

STAGE_ARITH
  → byte, word, dword 단위 산술 변형

STAGE_INTEREST
  → 0, -1, 127, 128, 255 같은 interesting value 삽입

STAGE_EXTRAS
  → dictionary token 삽입 또는 치환

STAGE_HAVOC
  → 여러 변이를 랜덤하게 조합

STAGE_SPLICE
  → 서로 다른 queue input을 잘라 붙임
```

여기서 중요한 점은 AFL의 mutation이 완전한 랜덤 변형 하나로만 이루어져 있지 않다는 것이다.  
AFL은 deterministic stage와 random stage를 순서대로 사용하면서 입력을 체계적으로 변형한다.

---

### 4-4. `add_to_queue()` — 새로운 testcase를 queue에 추가

`add_to_queue()`는 새로운 testcase를 AFL queue에 추가하는 함수이다.

<img src="assets/img/AFL/mut4.png" width="400" alt="AFL 실습 이미지">

이 함수는 새 `queue_entry`를 만들고, 입력 파일 이름과 길이, depth 같은 정보를 저장한 뒤 linked list 끝에 붙인다.

중요한 부분은 다음이다.

<img src="assets/img/AFL/mut4-1.png" width="400" alt="AFL 실습 이미지">

```text
queued_paths++
  → queue에 저장된 path 수 증가

pending_not_fuzzed++
  → 아직 fuzzing되지 않은 testcase 수 증가

last_path_time = get_cur_time()
  → 마지막으로 새 path를 찾은 시간 갱신
```

즉 AFL은 새로운 coverage를 만든 입력을 단순히 파일로 저장하는 것이 아니라, 내부 queue 구조에 등록해 이후 mutation의 재료로 다시 사용한다.

---

### 4-5. `save_if_interesting()` — 의미 있는 입력만 저장

Mutation으로 생성된 모든 입력이 queue에 저장되는 것은 아니다.  
AFL은 target을 실행한 뒤, 그 입력이 새로운 coverage를 만들었는지 확인한다.

이 판단을 담당하는 함수가 `save_if_interesting()`이다.

코드 흐름을 보면 먼저 `has_new_bits(virgin_bits)`를 호출한다.

<img src="assets/img/AFL/mut5.png" width="400" alt="AFL 실습 이미지">

새로운 coverage가 없으면 바로 `0`을 반환하고, 해당 입력은 queue에 저장되지 않는다.  
반대로 새로운 coverage가 있으면 queue 파일 이름을 만들고 `add_to_queue()`를 호출한다.

<img src="assets/img/AFL/mut5-1.png" width="400" alt="AFL 실습 이미지">

그리고 실제 입력 내용을 파일로 저장한다.

<img src="assets/img/AFL/mut5-2.png" width="400" alt="AFL 실습 이미지">

즉 `save_if_interesting()`은 mutation 결과를 평가하는 필터 역할을 한다.

```text
mutation으로 입력 생성
        ↓
target 실행
        ↓
has_new_bits()로 coverage 확인
        ↓
새로운 coverage가 없으면 버림
        ↓
새로운 coverage가 있으면 add_to_queue()
        ↓
out/queue에 저장
```

또한 timeout이나 crash가 발생한 경우에는 `virgin_tmout`, `virgin_crash`와 비교한 뒤 `out/hangs`, `out/crashes`에 별도로 저장한다.

---

### 4-6. 실습으로 확인한 Mutation 동작

<img src="assets/img/AFL/afl-mut.png" width="400" alt="AFL 실습 이미지">

`afl-fuzz` 실행 화면에서는 현재 mutation stage와 실행 상태를 확인할 수 있다.

```text
now trying     : 현재 mutation stage
stage execs    : 현재 stage 실행 횟수
total execs    : 전체 실행 횟수
exec speed     : 초당 실행 횟수
total paths    : queue에 저장된 path 수
unique crashes : 발견된 crash 수
```

이번 실습에서는 `total paths`가 1로 유지되었다.  
이는 초기 seed 외에 새로운 coverage를 만드는 입력이 아직 발견되지 않았다는 의미이다.

퍼징을 중단한 뒤 output directory를 확인했다.

<img src="assets/img/AFL/afl-mut2.png" width="400" alt="AFL 실습 이미지">
```bash
ls -al out/queue
ls -al out/crashes
```

`out/queue`에는 초기 seed가 저장되어 있었다.

```text
id:000000,orig:seed.txt
```

반면 `out/crashes`는 비어 있었다.  
이는 현재 target code가 crash를 유발하는 로직을 포함하지 않거나, 짧은 실행 시간 동안 crash input을 발견하지 못했기 때문이다.

정리하면 Mutation 흐름은 다음과 같다.

```text
queue에서 seed 선택
        ↓
stage별 mutation 수행
        ↓
target 실행
        ↓
has_new_bits()로 coverage 확인
        ↓
새로운 coverage면 save_if_interesting()
        ↓
add_to_queue()로 queue 등록
        ↓
out/queue에 파일 저장
```

따라서 Mutation은 AFL이 새로운 실행 경로를 찾기 위해 입력을 계속 변형하는 단계이고, `save_if_interesting()`은 그 결과가 저장할 가치가 있는지 판단하는 핵심 로직이다.
